# Sequence Diagrams: Asynchronous Flows with Kafka

## 1. Description

Sequence diagrams show the temporal interaction between components via the event bus (Kafka). Two critical flows are presented:

- **Infraction ingestion and detection flow** (with optional photo).
- **Extreme weather alert flow**.

## 2. Infraction Detection Flow (PlantUML)

```plantuml
@startuml
title Sequence - Speeding Infraction Detection

actor "IoT Sensor" as sensor
participant "IoT Ingest" as ingesta
participant "Kafka (topic: vehicle-sightings)" as kafka_in
participant "Violation Detector" as detector
participant "Speed Limit Cache" as cache
participant "Evidence Packager" as packager
participant "Fine Manager" as manager
participant "Identity Client (gRPC)" as id_client
participant "Identity & Registry" as identity
participant "PostgreSQL (tickets)" as db
participant "Kafka (topic: infraction-tickets)" as kafka_out
participant "Notifications (email)" as notif

sensor -> ingesta: POST /vehicle (license plate, speed, location, timestamp)
ingesta -> ingesta: normalizes format
ingesta -> kafka_in: publishes VehicleSighted

kafka_in -> detector: consumes VehicleSighted
detector -> cache: getSpeedLimit(locationId)
cache --> detector: 80 km/h
detector -> detector: 95 > 80 => violation
detector -> packager: notifyViolation(sighting, speedLimit)

packager -> manager: createTicket(licensePlate, locationId, speed, limit)
manager -> id_client: getOwnerEmail(licensePlate)
id_client -> identity: gRPC (with circuit breaker)
identity --> id_client: email@example.com
id_client --> manager: ownerEmail
manager -> db: save(ticket) [immutable]
db --> manager: ticketId
manager -> kafka_out: publishes InfractionTicketCreated
kafka_out -> notif: consumes (sends email)
notif --> notif: email sent

@enduml
```

## 3. Flow Explanation

- The sensor sends a sighting **without a photo** (only license plate, speed, location, timestamp).
- Ingest normalizes and publishes `VehicleSighted` to Kafka.
- `Violation Detector` consumes and evaluates against the cached limit.
- If there is a violation, `Evidence Packager` is activated, then `Fine Manager`.
- `Fine Manager` queries the vehicle registry (synchronously) to obtain the email.
- The immutable ticket is persisted in PostgreSQL.
- `InfractionTicketCreated` is published to Kafka.
- Notifications consumes and sends the email to the owner.

## 4. Extreme Weather Alert Flow (PlantUML)

```plantuml
@startuml
title Sequence - Extreme Weather Alert

actor "Weather Station" as station
participant "IoT Ingest" as ingesta
participant "Kafka (topic: weather-metrics)" as kafka_weather
participant "Environmental Monitoring" as env
participant "Extreme Weather Detector" as detector
participant "Kafka (topic: extreme-alerts)" as kafka_alerts
participant "Internal Notifications" as notif
participant "Traffic Authority" as authority

station -> ingesta: POST /weather (temp, humidity, rain, wind)
ingesta -> ingesta: normalizes
ingesta -> kafka_weather: publishes WeatherMetricsRecorded

kafka_weather -> env: consumes
env -> detector: checkThresholds(weather)
detector -> detector: rain 55mm/h > threshold 50mm/h
detector -> kafka_alerts: publishes ExtremeWeatherAlertTriggered

kafka_alerts -> notif: consumes alert
notif -> authority: internal notification (API/event)
authority --> notif: ACK
@enduml
```

## 5. Alert Flow Explanation

- The weather station sends data every 5 minutes.
- `Environmental Monitoring` consumes `WeatherMetricsRecorded`.
- `Extreme Weather Detector` evaluates thresholds (e.g., rain > 50 mm/h).
- If exceeded, `ExtremeWeatherAlertTriggered` is published.
- `Notifications` consumes the alert and sends it to the **Traffic Authority** (internal system, not social media).

## 6. Notes on Asynchronicity and Resilience

- **Producers**: Ingest, Law Enforcement, Environmental.
- **Consumers**: All microservices (each with its own consumer group).
- **Guarantees**:
  - Kafka with `acks=all` and replication (factor 3) prevents event loss.
  - Idempotency in consumers via `processed_event_ids` table (prevents duplicates).
  - Dead Letter Queue (DLQ) for events that fail after retries.
- **Back-pressure**: Ingest applies HTTP 503 if the Kafka producer cannot send (buffer full). Sensors must retry with backoff.

## 7. Event ID Example

Each event includes a unique `eventId` (UUID v7) generated by the producer. Consumers verify idempotency by storing these IDs (with TTL) in Redis.
